# FAQ

## Is Stirling PDF Space official?

No. Stirling PDF Space is independent and unofficial. It is a hosted workflow planner for people evaluating Stirling PDF operations. It does not replace official docs, upstream support, paid offerings, community channels, or legal terms.

## What does Stirling PDF do?

Public project material describes Stirling PDF as an open-source PDF platform with desktop, browser, self-hosted server, and private API modes. It includes 60+ tools for tasks such as editing, signing, redaction, conversion, OCR, compression, merge, split, and automation.

## Is the license simple MIT?

The public license file describes the project as open-core. Content outside named restricted directories is MIT, while listed directories such as proprietary, SaaS, engine, desktop, cloud, and portal areas may have their own license files. Teams should review the license boundary before packaging or redistributing modified builds.

## Does every feature have an API?

No. Official API docs note that some operations are frontend-only, while many backend operations are exposed through REST API and local Swagger UI. Verify the exact endpoint on a running instance before promising automation.

## When should I self-host?

Self-host when document privacy, internal access controls, API automation, storage policy, or network boundaries matter. Plan Docker or Kubernetes deployment, authentication, TLS, storage cleanup, backups, monitoring, and updates before exposing it to users.

## Where does the hosted planner fit?

Use https://stirling-pdf.space/ before production rollout to draft a workflow plan, pricing package, deployment checklist, and evidence path. Use official docs for exact current commands, configuration names, API schemas, and upstream release details.
